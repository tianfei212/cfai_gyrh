# Go 版本程序解耦与演示端重构设计

## 背景

当前 `gyrh-go-v2` 已经同时包含图片改写、背景库、Skill 管理、风格配置、历史记录、302 GPT Image、Wan、Gemini、Qwen 等能力。随着功能增多，后端入口装配、HTTP Handler、模型调用和数据库访问之间的边界开始变重；前端也存在页面路由、页面逻辑和大型全局样式混在一起的问题。

本次重构选择“完整分层，但保持 API 行为兼容”的方案。目标是在不破坏现有业务功能的前提下，重新梳理 Go 后端模块边界，并把前端拆成一体机演示入口和管理入口。

## 目标

- 新开分支完成开发，工作结束后更新 `CHANGELOG` 并提交代码。
- 后端按更明确的分层架构重组，做到功能模块单一、相似功能集中。
- 模型连接代码按实际模型 provider 聚合到对应目录。
- 所有迁移和新增函数提供中文注释，复杂流程提供中文说明。
- 后端补齐 `info`、`debug`、`error` 级别日志；`debug` 下可以查看详细请求和响应，包括头部与 payload，但必须脱敏密钥。
- 前端拆分页面、逻辑和皮肤样式文件。
- 当前 `/` 管理页面迁移为 `/admin_viewer`。
- 新建 `/` 一体机演示页面，复用当前页面能力，但隐藏“背景库、SKILL 管理、风格转换配置、退出”菜单。
- 演示页面面向 55 寸 2K 触控屏，按 16:9 大屏交互优化。

## 非目标

- 不改变现有 `/api/v1` 对外接口语义，除非发现现有接口存在明确缺陷且必须修复。
- 不把当前单体服务拆成多个独立进程或微服务。
- 不引入新的数据库系统。
- 不提交运行产物、数据库快照、生成图片或本地密钥文件。
- 不把演示端做成公网安全系统；本项目默认仍是展厅内网/一体机部署场景。

## 后端分层架构

后端采用清晰分层，建议目录如下：

```text
backend/internal/
  api/                 HTTP 路由、中间件、请求响应 DTO
  application/         用例编排：图片改写、背景管理、Skill 管理、风格管理
  domain/              核心实体与接口：Image、RewriteTask、BackgroundPrompt、Skill、ModelProvider
  infrastructure/      SQLite、存储、本地文件、OSS、外部模型客户端
  platform/            配置、日志、启动装配、进程生命周期
```

### `api`

`api` 只负责 HTTP 边界：

- 注册路由和中间件。
- 解析请求、校验基础参数、写出响应。
- 把请求转换为 application command/query。
- 统一处理错误响应和访问日志。

`api` 不直接访问 SQLite、文件系统、OSS 或模型客户端。

### `application`

`application` 负责业务用例编排，例如：

- 图片上传与生成图入库。
- 图片改写任务创建、执行、查询和 SSE 推送。
- 背景提示词导入、远端同步、Qwen 建议和英文同步。
- Skill 管理与当前激活模型提示词读取。
- 风格提示词管理。

这一层可以依赖 `domain` 接口，但不依赖具体 SQLite 或具体 HTTP 模型客户端实现。

### `domain`

`domain` 定义核心对象和接口：

- `Image`、`GeneratedImage`、`ReferenceImage`。
- `RewriteTask` 与任务状态。
- `BackgroundPrompt`。
- `Skill` 与 provider。
- `StylePrompt`。
- `ModelProvider`、`ModelComposeRequest`、`ModelComposeResult`。
- 仓储接口，如 `ImageRepository`、`RewriteTaskRepository`、`BackgroundPromptRepository`。

`domain` 不包含外部 HTTP 调用、不读配置文件、不处理 mux 请求。

### `infrastructure`

`infrastructure` 提供具体实现：

- SQLite 仓储实现。
- 本地文件和 OSS 存储实现。
- 外部模型客户端。
- 远端素材同步客户端。

模型包按实际连接的 provider 聚合：

```text
backend/internal/infrastructure/model/
  gemini/
  wan/
  qwen/
  302_gpt_image/
```

Go 包名不能以数字开头，因此 `302_gpt_image` 目录内包名建议使用 `gptimage302`。业务 provider 名保持 `302-gpt-image`，避免破坏前端和数据库中的既有数据。

### `platform`

`platform` 放置跨层基础设施：

- 配置加载与环境变量覆盖。
- 日志初始化与脱敏工具。
- 服务装配。
- 启动与关闭生命周期。
- 后台 watcher 或 bootstrap 流程。

`cmd/server/main.go` 只保留最薄的启动入口：加载 app、启动 HTTP 服务、处理退出信号。

## 后端日志设计

日志统一由 `platform/logging` 或同等包提供，至少支持：

- `Info`：关键业务节点。
- `Debug`：详细诊断信息。
- `Error`：失败和异常。

### `info` 日志

`info` 用于记录正常业务流转：

- 服务启动、监听地址、运行模式。
- 路由注册完成。
- 数据库连接和迁移完成。
- 图片改写任务创建。
- 当前选择的 provider。
- 模型任务提交成功。
- 生成结果保存完成。
- 后台同步任务开始和结束。

### `debug` 日志

`debug` 用于问题排查，允许打印详细请求和响应：

- HTTP method、url、query、headers、payload。
- 上游模型请求头和请求体。
- 上游模型响应状态、响应头和响应 body。
- DB 查询关键参数和返回数量。

必须脱敏以下内容：

- `Authorization`。
- API Key。
- Cookie。
- HMAC secret。
- 本地访问 token。
- 任何环境变量形式的密钥。

### `error` 日志

`error` 用于记录失败并带足上下文：

- 请求 ID 或任务 ID。
- provider。
- image id / background id / skill id。
- 上游状态码。
- 有限长度的错误响应 body。
- Go error chain。

错误日志不能吞掉原始错误，应保证上层可以返回合适响应或标记任务失败。

## 中文注释设计

所有迁移和新增的导出类型、导出函数必须使用中文注释说明职责。非导出但业务复杂的函数也必须有中文注释。

函数注释建议包含：

- 函数负责什么。
- 关键入参代表什么。
- 成功返回什么。
- 失败时如何处理。

复杂流程内部使用简短中文注释标明阶段，例如“保存前景图”“构造模型请求”“写入任务完成状态”。

## 前端路由设计

前端拆成两个入口：

- `/`：一体机演示入口。
- `/admin_viewer`：管理入口，承载当前完整页面。

推荐引入 `react-router-dom`。如果为了减少依赖，也可以用 `window.location.pathname` 做最小路由，但后续刷新恢复和页面扩展会更差，因此默认方案是使用 `react-router-dom`。

### `/` 演示入口

`/` 不需要登录，不展示以下菜单：

- 背景库。
- SKILL 管理。
- 风格转换配置。
- 退出。

演示入口保留面向观众的核心流程：

1. 进入首页。
2. 选择背景。
3. 拍摄或上传人物图。
4. 等待生成。
5. 预览和重新生成。
6. 查看有限历史或返回首页。

演示入口可以继续调用现有 API；如果接口仍要求签名，前端可使用现有演示环境配置自动加签，但用户不需要看到登录流程。

### `/admin_viewer` 管理入口

`/admin_viewer` 基本复制当前完整页面能力：

- 首页工作台。
- 历史记录。
- 背景库。
- SKILL 管理。
- 风格转换配置。
- 登录和退出。

管理员入口保留完整导航和管理能力。当前 `/` 下的页面行为迁移到 `/admin_viewer`，避免影响原管理功能。

## 前端目录与逻辑拆分

建议前端结构调整为：

```text
frontend/src/
  app/                 路由、入口装配、全局状态
  pages/
    kiosk/             "/" 演示入口页面
    admin/             "/admin_viewer" 管理入口页面
  features/
    capture/           拍摄与抠图逻辑
    preview/           预览与重新生成逻辑
    background/        背景选择和背景管理逻辑
    skill/             Skill 管理逻辑
    style/             风格配置逻辑
    history/           历史记录逻辑
  screens/             兼容迁移期的页面组件
  services/            API 请求和 rewrite task 订阅
  theme/
    tokens.css         颜色、字体、间距、圆角、阴影
    glass.css          玻璃拟态按钮和面板
    kiosk.css          55 寸 2K 演示端布局
    admin.css          管理端布局
```

页面组件只负责布局和状态展示。业务请求、数据转换、任务订阅、分页计算等逻辑放到 `features`、`services` 或 `utils` 中。

## 前端视觉设计

整体风格遵循“苹果 UI 简约设计、玻璃按键、3D 效果、阴影效果、适合一体机演示的清晰文字”。

### 大屏参数

- 目标屏幕：55 寸 2K 触控屏。
- 目标比例：16:9。
- 目标分辨率：`2560x1440`。
- 使用 `clamp()` 和相对单位适配相邻尺寸。
- 主要按钮最小高度建议 `72px`，核心行动按钮建议 `88px` 以上。

### 视觉风格

- 深色高级渐变背景。
- 半透明玻璃面板。
- 背景模糊和柔和高光。
- 大圆角。
- 多层柔和阴影。
- 轻微 3D 悬浮卡片。
- 高对比、清晰、适合远距离阅读的中文字体。
- 微交互以稳定为主，包括按钮按压、卡片浮起、生成中呼吸光效。

### 演示端布局

演示端应减少管理干扰，优先突出当前任务：

- 顶部展示品牌、当前步骤和简短提示。
- 中心区域展示背景选择、摄像头或生成结果。
- 底部或右侧放置大尺寸触控操作按钮。
- 生成中页面展示明确进度状态，避免观众误以为系统卡死。
- 不显示管理配置入口。

### 管理端布局

管理端保留完整能力，但视觉应与演示端统一：

- 左侧导航或浮动控制轨。
- 中心工作区。
- 管理面板使用玻璃拟态卡片。
- 表单和列表保持足够对比度。
- 高风险操作提供明确确认状态。

## 迁移阶段

### 阶段一：分支与基础设施

- 创建重构分支。
- 建立新分层目录。
- 引入统一日志封装和脱敏工具。
- 保留旧代码路径，先建立兼容适配层。

### 阶段二：后端分层迁移

- 瘦身 `cmd/server/main.go`，把装配迁移到 `platform`。
- 把配置与启动流程迁移到 `platform`。
- 把图片改写主流程迁移到 `application`。
- 把核心实体和接口迁移到 `domain`。
- 把 SQLite、存储、OSS、模型客户端迁移到 `infrastructure`。
- 保持现有路由和响应结构兼容。

### 阶段三：模型与日志补齐

- 按 provider 目录迁移模型客户端。
- 补齐模型请求和响应 debug 日志。
- 给关键函数补齐中文注释。
- 增加任务级上下文日志。
- 验证密钥不会出现在日志中。

### 阶段四：前端双入口

- 引入或实现路由层。
- 把当前完整页面迁移到 `/admin_viewer`。
- 复制并裁剪一份演示流程作为 `/`。
- 隐藏演示端不需要的管理菜单。
- 保持核心生成流程可用。

### 阶段五：样式与 2K 适配

- 抽离 `theme/tokens.css`。
- 抽离玻璃拟态、演示端、管理端样式。
- 清理页面内联样式。
- 针对 16:9 2K 屏调整字号、间距、按钮触控面积和预览区域比例。
- 完成 `CHANGELOG` 更新。
- 跑测试并提交代码。

## 测试计划

后端测试：

- Go 单元测试覆盖 `domain` 关键对象和状态转换。
- Application service 测试覆盖图片改写、任务状态、背景 Prompt 选择。
- Infrastructure 测试覆盖 SQLite repo、模型客户端请求构造、日志脱敏。
- Router 层测试确认原 `/api/v1` 接口仍可用。

前端测试：

- 工具函数测试保持通过。
- 路由测试确认 `/` 和 `/admin_viewer` 分别渲染正确入口。
- 演示端确认不出现背景库、SKILL 管理、风格转换配置和退出菜单。
- 管理端确认完整菜单仍存在。

人工验证：

- 运行后端服务。
- 运行前端开发服务。
- 在 16:9 视口检查 `/` 演示流程。
- 检查 `/admin_viewer` 管理流程。
- 触发一次 Wan 或 Gemini 生成流程。
- 触发一次 302 GPT Image 异步任务流程。
- 检查 debug 日志中请求响应完整但密钥已脱敏。

## 风险与控制

- 后端目录迁移范围大，必须分阶段迁移并保持 API 兼容。
- `ImageHandler` 当前职责过多，拆分时优先通过 application service 承接流程，避免边拆边改业务语义。
- 模型 provider 名称已经进入前端和数据库，业务 provider 名不应随目录迁移改变。
- 前端大型 CSS 拆分容易出现视觉回归，应先抽 tokens，再拆页面样式。
- `/` 无登录演示入口适合展厅内网，一旦公网部署需要重新评估鉴权策略。
- 当前 git 状态包含数据库、生成文件、二进制等运行产物，提交前必须排除。

## 完成标准

- 后端核心代码已按分层目录组织。
- 当前 `/api/v1` 接口保持兼容。
- 模型客户端按 provider 聚合。
- 新增和迁移函数具备中文注释。
- 关键流程具备中文 `info`、`debug`、`error` 日志。
- `/admin_viewer` 承载原管理页面。
- `/` 承载无登录演示页面，并隐藏管理菜单。
- CSS 和皮肤样式已从页面逻辑中抽离。
- 2K 16:9 触控屏视觉与交互完成定向优化。
- 自动测试和关键人工验证通过。
- `CHANGELOG` 已更新。
- 最终提交不包含密钥、数据库快照、生成图片或本地二进制产物。
